DuckDB SQL引擎源码浅析

本文介绍DuckDB的SQL引擎，主要包括Parser、Analyzer、Optimizer及Executor。这也是一个经典的SQL Engine所具备的主要部分。

一个查询的入口函数为 ClientContext::Query(const string &query, bool allow_stream_result)。

Parser

DuckDB的解析器是基于PostgreSQL的libpg_query语法解析库进行修改而来的。

主要修改如下：

  1. 输出改为C++语言
  2. parser和辅助数据结构用duckdb_libpgquery namespace括起来
  3. 将文件拆到多个小文件，而不是以前单独一个大文件
  4. 做了部分代码去重简化

DuckDB的parser入口函数为Parser::ParseQuery(const string &query)。

Optimizer

DuckDB的优化器是基于规则的优化器（即RBO），优化器入口函数为 Optimizer::Optimize函数。不同的规则在执行顺序上有依赖，需要小心安排。

逻辑计划遍历框架

首先需要描述下LogicalOperatorVisitor::VisitOperator的逻辑。

逻辑执行计划的优化，通常需要遍历逻辑执行计划树，然后对树的节点或者结构进行对应的优化，很多优化规则的执行逻辑都是从这个逻辑继承过来的。 这块逻辑也比较清晰，就是从根节点开始进行递归访问，并且每个operator的expression也都处理一遍。

Expression Rewrite

表达式重写规则包括多项优化规则，expression rewrite的执行框架如下

代码中目前有19项表达式重写规则，具体介绍如下：

  • ConstantFoldingRule（常量折叠）：1+1 优化为 2

  • DistributivityRule（分配律）：(X AND B) or (X AND C) OR (X AND D) 优化为 X AND (B OR C OR D)

  • ArithmeticSimplifcationRule（算术化简）：X * 0 优化为 0

  • CaseSimplificationRule：CASE WHEN 1=1 THEN x ELSE y END 优化为 x

  • ConjunctionSimplification：X OR FALSE 优化为 X

  • DatePartSimplicationRule：date_part(‘year’, x) 优化为 year(x)

  • ComparisonSimplificationRule：[x = NULL] 优化为 NULL

  • InClauseSimplificationRule （ cast(x) in list[a, b, c]场景，减少cast(x) ，不过unittest中的例子似乎没有满足优化条件）

  • EqualOrNullSimplicationRule：a=b OR (a IS NULL AND b is NULL) 优化为 a IS NULL DISTINCT FROM b

  • MoveConstantsRule：x + 1 = 5000 优化为 x = 4999

  • LikeOptimizationRule：regexp_matches(c, ‘^Prefix’) 优化为 LIKE ‘Prefix%’
